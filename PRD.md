# Product Requirements Document

## Confluence Awards — Augmented Red Carpet Experience

---

## Overview

A mobile-first web app that lets users take a selfie, choose an AI-powered visual effect, and receive a generated result (image or video) of themselves with that effect applied. The experience is branded for the Confluence Awards event.

**Branding:** "Confluence Awards — Augmented Red Carpet Experience"

**Tech stack:**

- Framework: Next.js (App Router), deployed on Vercel
- Styling: TailwindCSS
- Camera: `react-webcam`
- Carousel: embla-carousel-react
- Database: Supabase (Postgres)
- File storage: Vercel Blob (user photos, generated results)
- AI generation: fal.ai
- Background job orchestration: Trigger.dev

---

## Pages & Routes

| Route       | Page             | Description                             |
| ----------- | ---------------- | --------------------------------------- |
| `/shot/new` | Create Shot Page | Camera capture + effect selection       |
| `/shot/:id` | Read Shot Page   | View generation status and final result |

## Page 1: Create Shot Page (`/shot/new`)

This is a **single page** with three sequential states. The route does not change between states.

### State 1: Camera View — Default

**Figma:** `node-id=1-5`

The user sees a live camera feed filling the viewport.

**UI:**

- Full-screen camera viewport (rounded corners), powered by `react-webcam`
- Branding header: Confluence Awards logo + "AUGMENTED RED CARPET EXPERIENCE" subtitle
- Background: dark with swirling purple/blue gradient in the top portion
- Camera controls overlay at the bottom of the viewport:
  - **Capture button** (large white circle, centered-left) — takes the photo
  - **Flip camera button** (icon, centered-right) — toggles front/rear camera

**Behavior:**

- Camera stream is active on page load, defaulting to front-facing camera
- Tapping Capture takes a screenshot via `react-webcam`'s `getScreenshot()` and transitions to State 2

---

### State 2: Camera View — Accept/Reject

**Figma:** `node-id=5-13`

The captured photo is frozen in the viewport. The user decides to keep it or retake.

**UI:**

- Same layout as State 1, but the live camera feed is replaced with the frozen captured image
- Camera controls overlay changes to two buttons:
  - **Retry button** (white circle, undo/redo icon) — discards the photo
  - **Accept button** (white circle, checkmark icon) — confirms the photo

**Behavior:**

- Tapping Retry → returns to State 1, camera stream resumes
- Tapping Accept → transitions to State 3

**Transition (State 1 → State 2):** Instant freeze of camera frame; controls swap in place.

**Transition (State 2 → State 3):** The camera viewport/frozen photo element animates smoothly — shrinking and repositioning to become the accepted photo thumbnail in the upper-center area of the screen. The effects carousel simultaneously slides up from the bottom. This should be a single coordinated animation using CSS transitions or a layout animation library (e.g. Framer Motion).

---

### State 3: Select Effect

**Figma:** `node-id=8-41`

The user picks an AI effect to apply to their photo.

**UI:**

- Branding header (same as other states)
- **Accepted photo thumbnail** — the photo element from State 2, now animating into a small rounded thumbnail positioned upper-center. Has a white checkmark badge in the bottom-right corner.
- **Effects carousel** — horizontal scrollable row of large square preview images filling the lower portion of the screen. The center card is emphasized (active/selected).
- **"Choose Effect →" button** — white pill button overlaid on the center carousel card. Tapping it submits the shot.

**Behavior:**

- User scrolls horizontally through the carousel to browse effects
- The center card snaps into focus (snap scrolling)
- The centered effect is the selected effect
- Tapping "Choose Effect →":
  1. Uploads the captured photo to **Vercel Blob** via a Next.js API route, receives a `image_url`
  2. Calls `POST /api/shots` with `{ image_url, effect_id }`
  3. Receives `{ id }` in response
  4. Navigates to `/shot/:id`

**Effect data** (loaded on page mount): `GET /api/effects` returns list of `{ id, name, preview_image_url }`

## Page 2: Read Shot Page (`/shot/:id`)

This is a **single page** with two states depending on whether the result is ready. The route is `/shot/:id`.

### State 1: Pending

**Figma:** `node-id=9-59`

The shot is being generated by the backend.

**UI:**

- Branding header
- Two thumbnails side by side (upper area):
  - Left: **Accepted photo thumbnail** (portrait crop, smaller) with white checkmark badge
  - Right: **Accepted effect thumbnail** (square, larger) with white checkmark badge
- **"Generating..."** text below the thumbnails, centered, white

**Behavior:**

- Polls `GET /api/shots/:id` every 3 seconds
- When `result_url` is non-null in the response, transitions to State 2

**Transition (Pending → Shot Display):** Thumbnails fade out; the result media fades/scales in.

---

### State 2: Shot Display

**Figma:** `node-id=9-106`

The final generated result is ready.

**UI:**

- Branding header
- **Media display** — full-width, tall rounded rectangle filling most of the screen:
  - If `result_type === "video"`: render an autoplaying, looping `<video>` element
  - If `result_type === "image"`: render an `<img>` element
- **"Share →"** pill button overlaid at the bottom of the media

**Behavior:**

- Media autoplays (if video) on transition from Pending state
- Tapping "Share →" invokes the **native Web Share API** (`navigator.share()`), passing the result URL. No Instagram-specific deep linking — the user chooses their share destination via the OS share sheet.

## Backend Architecture

### Request flow

```
Client
  → POST /api/shots (Next.js API route)
      → insert row into Supabase (result_url: null)
      → trigger Trigger.dev job
      → return { id }

Trigger.dev job (background, long-running)
  → step 1: call fal.ai API #1 (2–5 min)
  → step 2: call fal.ai API #2 (2–5 min)
  → step 3 (if needed): call fal.ai API #3
  → upload final result to Vercel Blob
  → update Supabase row: result_url, result_type

Client (polling /shot/:id)
  → GET /api/shots/:id (Next.js API route)
      → read row from Supabase
      → return shot record
```

Each Trigger.dev step is independently retried on failure. Total job duration can exceed Vercel function time limits without issue.

### Image upload flow

Before `POST /api/shots` is called, the client uploads the captured photo:

```
Client → POST /api/upload (Next.js API route)
           → stream to Vercel Blob
           → return { image_url }
```

## API Contract

### Upload

```
POST /api/upload
Body: image file (multipart/form-data)
→ 200 { image_url: string }   // Vercel Blob URL
```

### Effects

```
GET /api/effects
→ 200 { effects: [{ id, name, preview_image_url }] }
```

### Shots

```
POST /api/shots
Body: { image_url: string, effect_id: string }
→ 201 { id: string }
```

```
GET /api/shots/:id
→ 200 {
    id: string,
    image_url: string,
    effect_id: string,
    result_url: string | null,       // null while pending; Vercel Blob URL when done
    result_type: "image" | "video"   // indicates how to render result_url
  }
```

The client derives pending/complete status from `result_url`: if `null`, the shot is still generating.

## Data Model

Hosted on **Supabase** (Postgres). File assets (photos, generated results) are stored on **Vercel Blob**; only URLs are stored in the database.

### `effects` table

| Column              | Type | Notes                             |
| ------------------- | ---- | --------------------------------- |
| `id`                | uuid | Primary key                       |
| `name`              | text | Display name                      |
| `preview_image_url` | text | Vercel Blob URL; used in carousel |

### `shots` table

| Column        | Type      | Notes                                                   |
| ------------- | --------- | ------------------------------------------------------- |
| `id`          | uuid      | Primary key                                             |
| `image_url`   | text      | Vercel Blob URL of uploaded user photo                  |
| `effect_id`   | uuid      | FK → effects                                            |
| `result_url`  | text      | Vercel Blob URL of generated result; null while pending |
| `result_type` | enum      | `image`, `video`                                        |
| `created_at`  | timestamp |                                                         |

## State Transition Summary

```
[/shot/new - Camera Default]
  → (capture) → [Camera Accept/Reject]
    → (retry)  → [Camera Default]
    → (accept) → [Select Effect]  ← photo animates smoothly into thumbnail
      → (choose effect) → POST /api/shots → navigate to /shot/:id

[/shot/:id - Pending]  ← polls every 3s
  → (result_url non-null) → [Shot Display]
    → (share) → navigator.share() → OS share sheet (out of scope)
```
